GFW如何干掉数据包
2026-03-16 10:31:12

访问被拦截

在国内互联网输入 google.com 无法访问，是因为GFW干掉了数据包

源 MAC 目标 MAC 源 IP 目标 IP 源端口 目标端口 请求内容
00:1A:2B:3C:4D:5E 00:5F:6A:7B:8C:9D 192.168.1.100 8.8.8.8 49152 53 告诉我Google的IP地址

从物理网卡出去，在国内互联网中各路由器辗转，最后到出国的接口，GFW是五层设备解析到应用层检查流量的内容，它知道你要DNS解析，获得哪个网站的IP地址，这时候就可以干掉数据包，假设它没有对流量进行检查

数据包在国际互联网中各路由器辗转，最终达到DNS服务器，得到IP地址时回复一个数据包

源 MAC 目标 MAC 源 IP 目标 IP 源端口 目标端口 请求内容
00:5F:6A:7B:8C:9D 00:1A:2B:3C:4D:5E 8.8.8.8 192.168.1.100 53 49152 Google的IP地址9.9.9.9

DNS污染、劫持

数据包经过防火墙，检测到你访问国外网站，篡改数据包里的内容，又在国内互联网上辗转，最后到浏览器，看到 Google 的IP地址时 9.6.6.6 ，但是谷歌没对应IP地址的服务器，无法访问

绕过被拦截00x1

访问网站并不是直接发送DNS请求，检查本地的host文件，假设添加了对应的映射关系，实现不发送DNS请求就得到IP地址，应用层的https协议基于TCP，发送https的请求先使用TCP的方式和服务器建立连接

源 MAC 目标 MAC 源 IP 目标 IP 源端口 目标端口 请求内容
00:1A:2B:3C:4D:5E 00:5F:6A:7B:8C:9D 192.168.1.100 Google IP地址 49152 443 我要和你连接

从物理网卡发出去，在国内互联网辗转，主机不在国内又经过GFW，它解析到网络层就知道你要访问的IP地址，发现对应IP地址在黑名单里，干掉了数据包

TCP重置攻击

伪装成服务器返回一个数据包，篡改响应内容，从而终止TCP连接

源 MAC 目标 MAC 源 IP 目标 IP 源端口 目标端口 响应内容
00:1A:2B:3C:4D:5E 00:5F:6A:7B:8C:9D 192.168.1.100 GFW伪装IP 49152 443 滚

绕过被拦截00x2

谷歌的服务器不止一个IP，假设是 9.9.9.1 经过防火墙发现IP地址不在黑名单，放行，服务器返回响应，成功建立连接

源 MAC 目标 MAC 源 IP 目标 IP 源端口 目标端口 请求内容
00:1A:2B:3C:4D:5E 00:5F:6A:7B:8C:9D 192.168.1.100 142.250.190.78 49152 443 把Google首页的数据给我

虽然是网络层的IP不在黑名单里，但它能解析到应用层，需要Google.com的首页数据，但是域名也存在黑名单，又给数据包干掉了

绕过被拦截00x3

国外vps代替访问网站，假设监听10086端口

源 MAC 目标 MAC 源 IP 目标 IP 源端口 目标端口 请求内容
00:AA:BB:CC:DD:EE 00:FF:11:22:33:44 203.0.113.10 国外vps 49152 10086 帮我访问谷歌

GFW解析到应用层里的内容是让VPS帮你访问网站，又干掉了，把VPS的IP也加进黑名单里

归根结底，还是被发现了意图

使用VPN对数据进行加密，但是特征十分明显，虽然看不到你具体要干啥，但知道你在建立VPN连接，但它不止翻墙这一个功能也可以用来访问内网，放行
把IP加入监控，当和这台VPS进行大流量的连接，就会干掉连接

当然它还会主动探测，向服务器发送请求，只不过前提是翻墙协议存在漏洞
